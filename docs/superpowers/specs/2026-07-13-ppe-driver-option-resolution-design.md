# PPE 駕駛選項解析修正設計

## 目標

修正加油紀錄在當月沒有同一位駕駛既有資料列時出現 `missing driver` 的問題，並檢查、強化同屬 PPE 車輛系統的車輛里程駕駛代碼解析，避免姓名與代碼不一致或缺少代碼仍繼續儲存。

## 已確認根因

PPE 加油明細頁將駕駛選項放在 Kendo DropDownList 的 `dataSource`，格式為 JSON `Text / Value` 項目。中文姓名在頁面原始程式碼內使用 Unicode 跳脫，例如「郭國偵」為 `\u90ED\u570B\u5075`。

現行程式直接在原始程式碼內搜尋中文字；搜尋失敗後，才從當月既有加油資料列反查駕駛代碼。因此，當月既有列恰好包含同一位駕駛時會成功，沒有同一位駕駛時就回傳 `missing driver`。延長固定等待時間不會改變原始程式碼的 Unicode 表示方式，不能解決根因。

## 範圍

修改公務電腦來源套件：

## 駕駛選項解析

建立可單元測試的共用解析邏輯：

1. 從選項物件精確讀取顯示姓名與代碼，不使用 `JSON.stringify(...).includes(...)` 模糊搜尋。
2. 顯示姓名支援 `Text`、`Name`、`DriverName`、`UserName`、`EmpName` 等 PPE 已知欄位。
3. 代碼支援 `Value`、`Id`、`UserId`、`EmpId`、`Code`、`Driver` 等已知欄位。
4. 姓名只做前後空白及連續空白正規化後的精確比對，不使用部分姓名或模糊包含，以免選錯人。
5. 代碼為空或零時視為無效，不得繼續儲存。

## 加油紀錄流程

加油頁會讀取 `document.scripts`，擷取同時含有 `Text` 與 `Value` 的有效 JSON 選項物件，再交由共用解析邏輯比對。JSON 解析會自然還原 `\uXXXX`，因此可正確取得「郭國偵」對應的 `2448`。

流程會輪詢等待指定駕駛選項可取得，最長沿用現有明細頁等待上限 12 秒。這是條件式等待，不額外堆疊固定睡眠。取得代碼後才填入 Kendo 資料列；當月既有資料列只能作為相同姓名且已有有效代碼時的相容備援，不再是主要來源。

若等待後仍找不到，錯誤必須包含請求姓名及已取得的候選姓名摘要，方便判斷是人員清單尚未更新或頁面結構改變，不輸出帳密或其他敏感資料。

## 車輛里程流程

車輛里程目前從已解碼的 `window.driverList` 取資料，沒有加油頁的 Unicode 原始字串問題，但現行 `JSON.stringify(...).includes(driverName)` 是模糊搜尋，而且找不到時可能沿用新資料列原本的 `Driver`。

本次改為：

- 讀取 `window.driverList` 後使用同一個精確解析邏輯。
- 只有資料列既有 `DriverName` 與請求姓名精確相同、且 `Driver` 有效時，才允許沿用既有代碼。
- 新增列若無法解析有效駕駛代碼，應在填寫階段停止並回報 `missing vehicle mileage driver`，不得只寫姓名後儲存。
- 現有已成功的車輛選擇、里程計算、儲存及確認流程保持不變。

## 後台診斷

`missing driver`、`missing fuel driver` 與 `missing vehicle mileage driver` 必須在通用登入字樣判斷之前分類，避免事件明細前綴「登入帳號」造成誤判。

- 加油：未完成點顯示「填寫加油紀錄」，原因說明駕駛清單找不到指定人員或代碼。
- 里程：未完成點顯示「填寫返隊時間與里程」，原因同樣指出駕駛選項解析失敗。
- 下一步建議確認 PPE 人員清單是否包含該駕駛；程式修正後可由任務頁單獨重試對應站點。

## 測試條件

測試先失敗再實作，至少涵蓋：

- Unicode 跳脫的 `Text` 可解析成中文姓名及正確 `Value`。
- `Text / Value` 欄位順序不同仍可解析。
- 姓名外圍或連續空白可正規化後精確匹配。
- 部分姓名不得誤配。
- 無既有加油資料列時仍可取得駕駛代碼。
- 加油頁找不到指定人員時回報候選摘要，不誤報登入失敗。
- 車輛里程從 `window.driverList` 精確取得代碼。
- 車輛里程找不到代碼時停止，不得沿用其他駕駛的代碼。
- 原有加油、里程及完整測試套件保持通過。
