# 救護後台保存、案件類別與更新流程設計

## 目標

一次修正五項已確認問題：救護後台案件資料意外歸零與七日保存、後台成功／失敗篩選、NAS 登打頁返回首頁、`其他-打撈浮屍` 案件支援、登打明細時間顯示，以及公務電腦更新黑窗等待按鍵。

## 現況證據與根因

- 2026-07-10 讀取 live NAS `artifacts/public_pc/task_events.json`，檔案實際只有 1 件、1 個唯一 `task_id`，因此「最多 100 件」不是本次只剩一件的直接原因。
- `public_pc_reports()` 遇到 I/O 或 JSON 解析錯誤時直接回傳空清單；下一次 `upsert_public_pc_report()` 會把這個空清單寫回，只留下最新案件。這是可確定會清空既有歷史的錯誤路徑。
- 公務電腦可在短時間回報同一或不同案件的多個事件，現有讀取、合併、寫入沒有鎖，且原子寫入使用固定 `.tmp` 名稱，並行寫入可能互相覆蓋。
- `UPDATE_PACKAGE.bat` 的三個失敗分支仍有 `pause`，因此只要更新或自我修復回傳非 0，黑窗就會等待按鍵。

## 設計

### 1. 救護後台七日保存與資料保護

- 所有狀態的案件依 `updated_at` 統一保留七天，不再以 100 件截斷。
- 對 `task_events.json` 的讀取、合併、寫入加上同一把程序內鎖。
- 每次成功寫入同時更新 `task_events.backup.json`。
- 主檔遺失或損壞時讀取備份；主檔與備份都損壞時，回報寫入必須失敗，禁止把空清單覆蓋回去。公務電腦現有 pending queue 會保留未確認回報並重試。
- 後台預設顯示全部；上方提供「成功案件」「失敗案件」分類及「顯示全部」重設。部分失敗歸失敗，執行中只出現在全部。

### 2. NAS 登打頁返回首頁

- `/app` 依 request host 判斷：NAS 顯示連到 `/` 的「返回首頁」，本機 `127.0.0.1:8090/app` 不顯示。
- 不改動既有單頁登打流程，也不改動任務明細頁既有導覽。

### 3. `其他-打撈浮屍` 案件

- 案件查詢列除了緊急救護與火災，也納入完整類別 `其他-打撈浮屍`。
- 匯入表單時將事由映射為既有選項「溺水」。
- 此類別視同救護案件，因此符合四名以上出勤人員時仍可顯示兩車登打。
- 工作紀錄簿維持既有固定「勤務項目：救護」，並由匯入後的 `case_reason` 寫入「事由：溺水」。

### 4. 救護後台登打明細

- 單車標題由「車輛」改成「登打明細」；兩車標題維持「1車」「2車」。
- 每行順序調整為：車輛、傷病患、出勤時間、返隊時間、司機、里程、耗材、消毒項數。
- 出勤時間使用案件 `case_time`；兩車共用同一出勤時間，各自顯示自己的返隊時間。

### 5. 公務電腦更新黑窗

- 從建包來源與目前公務電腦包的 `UPDATE_PACKAGE.bat` 移除所有 `pause`。
- 成功與失敗皆以原本 exit code 自動結束，不再等待按鍵。
