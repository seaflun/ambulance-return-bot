# 公務電腦舊版儲存誤判調和設計

## 目標

修正公務電腦在 `2026.07.13.2127` 舊版執行時，因官方網站沒有成功提示而留下的錯誤失敗狀態，並把修正後的同一筆任務重新回報 NAS `/admin/public-pc`。真正含有明確錯誤的案件不得被改成成功。

## 已確認問題

- 公務電腦本機任務與 NAS `public_pc/task_events.json` 是兩份資料；NAS 只會在收到相同 `task_id` 的新事件時更新卡片。
- 本機「人工確認」目前只更新 `JsonTaskStore`，沒有呼叫 `report_public_pc_task_event()`，因此 NAS 卡片不會同步。
- 更新 Worker 只改變後續執行邏輯，不會回頭重算舊任務。
- 目前五筆失敗卡中，三筆只包含舊版靜默儲存誤判；另外兩筆分別是加油缺少駕駛與耗材車輛比對失敗，必須維持失敗。

## 安全邊界

- 只接受下列已知舊版訊息與對應站別，不用模糊關鍵字推測：
  - 工作：已按下儲存，但未收到儲存成功回應。
  - 里程：已填寫並按下儲存，但未偵測到確認視窗。
  - 消毒：已更新項目，僅缺少 `save response` 確認。
  - 耗材：`耗材儲存未取得明確成功回應：未出現確認訊息`。
- 任何其他失敗原因、例外類型、案件或車輛不符、缺少駕駛、頁面逾時、部分寫入或伺服器明確錯誤都不調和。
- 不刪除舊事件；新增一筆「舊版誤判已修正」稽核事件，保留原始時間線。
- 不重新開啟或送出任何官方表單。
- 憑證、Chrome profile、任務 JSON 與 NAS artifacts 不加入 Git 或發行包外的追蹤檔。

## 設計

### 1. 原子且可重複執行的本機調和

在 `JsonTaskStore` 增加單一調和操作。它在同一把 store lock 內讀取任務、逐站比對精確舊版特徵、把符合的狀態改成對應的 `*_saved`，並清除舊的錯誤指引欄位。

若調和後所有應執行站別皆成功，整體狀態改為 `desktop_fast_completed`，worker queue 維持或轉為 completed。操作追加一筆稽核事件後原子寫回。再次執行時因舊狀態已不存在，不得重複修改或重複新增事件。

### 2. 公務電腦啟動時回報舊任務

只在 `public_pc_reporting_enabled()` 的公務電腦模式啟動一次背景調和：掃描仍在本機保存期內的近期任務，呼叫上述原子操作。每筆有變更的任務都使用原 `task_id` 呼叫 `report_public_pc_task_event()`，讓 NAS 透過既有 upsert 覆蓋目前 `site_statuses`、`overall_status`、`last_action` 與 `last_status`，同時保留歷史 events。

NAS 暫時無法連線時，沿用既有 pending report 機制排隊，不能回滾已完成的本機調和，也不能阻止本機網頁啟動。

### 3. 人工確認同步

`/tasks/<task_id>/sites/<site_key>/complete` 在 `store.mark_site_completed()` 成功後，取得更新後 payload 並呼叫 `report_public_pc_task_event()`。如此未來任何需要人工核對的站別，在本機按下確認後都會刷新 NAS 同一張卡片。

### 4. 顯示與稽核

調和成功後，NAS 卡片顯示最新成功狀態與「舊版無提示儲存誤判已修正」動作。展開完整事件時仍能看到原失敗訊息以及後續修正事件，避免抹除稽核證據。

## 測試

- 先建立會失敗的回歸測試，證明四種精確舊版訊息目前不會被調和或回報。
- 驗證四種已知誤判可轉成 `*_saved`，全部完成時整體轉為 `desktop_fast_completed`。
- 驗證加油缺少駕駛、耗材車輛不符、混合「靜默誤判＋明確錯誤」都保留失敗。
- 驗證重複啟動不重複事件、不重複回報未變更任務。
- 驗證 NAS 離線時使用 pending report，且本機啟動不失敗。
- 驗證人工確認 route 會回報更新後 payload；失敗或衝突時不回報。
- 完成 targeted tests、完整 unittest、Python compile、`git diff --check` 與程式碼審查。

## 發布與驗證

1. 從 `WinPython_公務電腦使用包` 建立新版本，明確 stage、commit、push 並發布 GitHub release。
2. 核對遠端版本檔、ZIP 內 `VERSION.txt` 與 SHA256。
3. 透過既有 NAS 遠端更新命令更新實際公務電腦並確認安裝版本。
4. 公務電腦重啟後確認三筆舊版靜默誤判卡已用原 task ID 更新；目前頁面預期由成功 8／失敗 5 變成成功 11／失敗 2。
5. 確認仍失敗的兩筆保留明確原因，不被調和。

若實際公務電腦已缺少某筆本機 task JSON，啟動調和必須略過並留下可讀日誌，不得憑 NAS 卡片內容猜測 task ID；該筆改走後續具身分驗證的人工修正流程。
