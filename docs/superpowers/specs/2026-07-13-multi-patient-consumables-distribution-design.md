# 同案多患者耗材頁自動分配設計

## 目標

當同一趟案件因多名患者產生多張一站通耗材頁，且這些頁面的 TEMSISID 主體相同、只有末兩碼患者序號不同時，公務電腦 worker 應辨識所有屬於任務車輛的患者頁，將網頁 APP 中該車的耗材分配到各頁，逐頁儲存並確認。

成功條件是同車的每張患者耗材頁都至少寫入一項耗材，並在一站通顯示為已確認。正常單患者流程與既有一案兩車流程不得退化。

## 現況與問題

- 現行流程先以案件日期、時間、地址或案件編號篩選耗材候選，再以車輛名稱或 PPE 車牌尋找唯一明細頁。
- 多患者案件可能產生例如 `...08031901`、`...08031902` 的兩筆 TEMSISID。兩頁屬於同一案件、同一車輛，只差末兩碼患者序號。
- 現行程式遇到兩筆案件證據相同、車輛也相同的候選時，無法選出唯一頁面，為避免寫錯紀錄而停止。
- NAS 後台目前會把這類「找不到符合車輛的唯一紀錄」歸為通用頁面元素錯誤，顯示成按鈕或欄位與預期不同，未反映真正原因。

## 名詞與辨識規則

- `完整 TEMSISID`：一站通耗材內容連結中的 `emmTemsisid` 值。
- `患者序號`：完整 TEMSISID 最後兩碼，例如 `01`、`02`、`03`。
- `TEMSISID 主體`：移除最後兩碼患者序號後的其餘字串。
- `同案多患者群組`：案件日期、時間、地址等案件證據相符，TEMSISID 主體相同，且明細頁實際車輛相同的一組候選頁。

只有末兩碼為兩位數患者序號的 TEMSISID 才能參與自動分組。格式異常、患者序號重複或案件證據不一致時，不得猜測或自動儲存。

## 設計

### 1. 先分車，再分患者

耗材候選處理順序固定如下：

1. 以案件日期、時間、地址與案件編號等既有證據篩選候選頁。
2. 開啟候選明細，從頁面文字及表單欄位讀取實際車輛名稱或 PPE 車牌。
3. 先按實際車輛分組。
4. 在同一車輛群組內，再按 TEMSISID 主體分組。
5. 將患者頁依末兩碼數字由小到大排序。

若同一案件由新坡92與新坡93共同出勤，兩車必須使用各自在網頁 APP 中填寫的耗材，分別處理自己的患者頁，不得跨車合併或平分。

範例：

| 車輛 | 患者頁 | 耗材來源 |
|---|---|---|
| 新坡92 | `01`、`02` | 1車／新坡92耗材 |
| 新坡93 | `03`、`04`、`05` | 2車／新坡93耗材 |

### 2. 耗材分配規則

對同車群組內的每一種耗材，使用患者頁數做整數平均分配：

- 每頁先取得 `數量 // 頁數`。
- 餘數依患者序號由小到大，每頁再增加一個，直到餘數用完。
- 數量為零的耗材不寫入該頁。
- 正常分配時，各頁數量加總必須等於網頁 APP 中該車的原始耗材總量。

例如手套3雙分配到 `01`、`02`：`01=2`、`02=1`。只有1件的品項分配到兩頁時，寫入 `01=1`、`02=0`。

### 3. 空白頁補助手套

完成正常平均分配後，若任何患者頁仍未分到任何耗材，該頁補入：

`桃-9吋手套-L(雙) ×1`

這是唯一允許增加原始耗材總量的例外，目的僅是讓每張患者頁至少有一項耗材並完成已確認。不得替已有耗材的頁面額外補手套，也不得用其他耗材補空白頁。

### 4. 逐頁儲存與讀回

每張患者頁依序執行：

1. 開啟指定 TEMSISID 明細。
2. 再次確認該頁實際車輛屬於目前車輛群組。
3. 清除既有耗材列。
4. 填入該患者頁分配到的耗材。
5. 儲存並完成確認。
6. 讀回品項與數量，與預期分配逐項比對。

同車所有患者頁都通過讀回驗證，該車耗材才算完成；任務中所有車輛的患者頁都完成，耗材站才回報成功。

### 5. 重跑與部分失敗

- 單獨重跑耗材時，重新取得候選頁並重新計算同一份確定性分配結果。
- 每頁在寫入前先清除舊資料，因此重跑不得累加耗材。
- 若前幾頁成功、後一頁失敗，保留各患者序號的成功／失敗紀錄。
- 重跑時可以安全地再次寫入已成功頁面，確保所有頁面最後符合相同分配規則。
- 車輛無法辨識、同案群組不唯一、TEMSISID 格式異常或讀回不一致時，停止該車耗材流程，不得選第一筆或猜測。

### 6. 後台狀態與可觀測性

成功紀錄應包含：

- 辨識到的車輛與患者頁數。
- 每個患者序號分配到的耗材與數量。
- 是否有空白頁補助手套。
- 每頁儲存與讀回確認結果。

成功訊息範例：

`辨識新坡93同案2位患者；01填入5件、02填入5件，兩頁均已儲存確認。`

補助手套訊息範例：

`03原分配為空，已補手套×1以完成確認。`

失敗時 NAS 後台應顯示「同案多患者耗材分配／確認失敗」，並列出總頁數、已成功頁面、失敗患者序號與原始原因，不再顯示成籠統的頁面按鈕錯誤。

## 相容性與非目標

- 單車單患者仍走現有單頁流程。
- 一案兩車仍使用現有 `vehicle_requests()` 所提供的每車任務與每車耗材；本功能只在各車內增加患者頁分配。
- 支援任意正整數患者頁數，不限制為2或4人。
- 本功能不修改網頁 APP 表單，不新增患者人數欄位；患者頁數以一站通實際候選頁為準。
- 本功能不自動改動里程、勤務工作紀錄或消毒的多患者行為。
- 本功能不取消既有的人工作業與保護性停止邊界。

## 測試案例

1. 單車單患者：維持既有選頁、填寫、儲存與訊息。
2. 單車兩患者：相同 TEMSISID 主體、相同車輛，耗材依 `01`、`02` 平分。
3. 奇數數量：手套3雙分配為 `01=2`、`02=1`。
4. 單項數量小於頁數：只有前段患者頁取得該品項。
5. 空白患者頁：只對空白頁補手套×1。
6. 單車五位以上患者：依任意頁數分配，所有頁面至少一項耗材。
7. 一案兩車，分別2位與3位患者：先按車輛分組，再分患者，兩車耗材完全隔離。
8. 同一 TEMSISID 主體混合不同車輛：各車只取得自己的候選頁。
9. 無法辨識車輛：停止儲存並回報明確原因。
10. TEMSISID 尾碼格式異常或重複：停止儲存並回報明確原因。
11. 中途某頁儲存失敗：保存各頁結果，耗材站回報部分失敗。
12. 失敗後重跑：清除並重填，不重複累加，最後讀回全部吻合。
13. 診斷分類：同案多患者錯誤不得再映射成頁面按鈕或欄位錯誤。

## 驗證與發布

- 先為候選分組、車輛分區、分配、補助手套、部分失敗與重跑建立會失敗的回歸測試。
- 執行耗材、runner、worker、web／後台 targeted tests，再執行 Python 編譯與完整 unittest。
- 從 `WinPython_公務電腦使用包` 修改來源並重建公務電腦套件與 `UPDATE\NAS包`；不得直接修改生成的 NAS 輸出。
- 更新版本、commit、push、發布 GitHub Release，驗證遠端版本檔、zip 內 `VERSION.txt` 與 SHA256。
- 部署 NAS 後重啟 `ambulance-app-1`；更新公務電腦 worker 後，使用單患者、同車雙患者及兩車多患者測試案件驗證實際頁面結果。
