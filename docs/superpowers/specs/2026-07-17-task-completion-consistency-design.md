# 登打任務全站完成一致性設計

## 目標

讓 App 登打任務頁、App 首頁最近任務、NAS 公務電腦後台與救護 Worker 使用同一套全站完成判斷，並修正下列問題：

- 單站登打成功不能提前把整筆任務判成完成。
- 四站流程部分失敗後，逐站補打至全部成功時，整筆任務必須轉成「四站登打完成」。
- 有啟用加油紀錄時，整筆任務必須改成「五站登打完成」。
- 全部有效站別完成後，任務頁的「四站／五站登打啟動」按鈕必須自動消失。
- 操作事件仍保留「單站補打成功：站別」，不得用整體完成文字覆蓋實際操作歷程。

## 名詞與狀態層級

系統明確區分三個層級：

1. **站別結果**：工作、里程、加油、耗材、消毒各自的執行結果，例如「消毒：完成」。
2. **操作事件**：使用者或流程這次做了什麼，例如「單站補打成功：消毒」。
3. **任務整體**：目前所有有效站別是否完成，例如「四站登打完成」。

站別結果與操作事件保留稽核用途；任務整體只描述目前狀態。任務整體完成不能由單次操作成功直接推定。

## 共用全站完成快照

在任務資料層提供單一、純粹的全站完成計算，所有顯示與狀態轉換都使用同一份結果。快照至少包含：

- `active_site_keys`：本任務真正需要執行的站別。
- `site_count_label`：`四站` 或 `五站`。
- `total_count`：有效站數。
- `completed_count`：已完成站數。
- `remaining_site_keys`：尚未完成站別。
- `failed_site_keys`：失敗站別。
- `waiting_site_keys`：待人工確認或需更新站別。
- `all_complete`：是否所有有效站別都已完成。

有效站別規則：

- 未啟用加油紀錄：工作、里程、耗材、消毒，共四站。
- 啟用加油紀錄：再加入加油，共五站。
- 已存在待人工確認的加油資料仍視為有效站，不能因後續設定差異而從完成判斷消失。

完成狀態只接受 `completed_by_user`、既有成功站別狀態及 `*_saved`。`not_started`、執行中、失敗、待人工確認、需更新、缺少或格式錯誤的站別都不得算成完成。

## 原子狀態調和

每次站別狀態寫入後，在同一把任務 store lock 內重新計算全站完成快照，包括：

- 本機單站登打結果。
- 本機四／五站快速執行結果。
- Worker 回報的站別結果。
- 人工確認完成。
- 任務編輯後產生的需更新站別。

狀態轉換規則：

- `all_complete = false`：不得把整體狀態保留或改成 `desktop_fast_completed`。
- `all_complete = true`：整體狀態統一為 `desktop_fast_completed`，worker queue 結束，並記錄一次全站完成轉換。
- 已經是全站完成時重複收到相同或延遲回報，不得重複新增全站完成事件或通知。
- 完成後編輯任務，只要任一有效站變成需更新，整體完成立即失效，登打按鈕重新出現。

單站操作本身只能寫入該站結果與單站操作事件。是否升級成四／五站完成，交由上述原子調和決定。

## 任務編輯後精準補登

任務編輯時比較正規化後的前、後資料，產生一份結構化的 `pending_edit_impact`。它只記錄欄位代號、中文欄位名稱、受影響站別及車輛識別，不在頁面摘要或事件中顯示完整舊值與新值。既有 `changed_sites_for_task_edit` 改由同一份影響分析取得站別集合，避免「畫面顯示要補登的站」與「實際被標成需更新的站」使用兩套規則。

欄位與站別映射如下：

| 編輯資料 | 受影響站別 |
| --- | --- |
| 案件日期、案件時間 | 工作、里程、耗材、消毒 |
| 案件地址 | 工作、里程 |
| 車號 | 該車所有有效站別；未啟用加油時不納入加油 |
| 駕駛 | 該車的工作、里程、加油 |
| 返隊日期、返隊時間、里程 | 該車的里程 |
| 加油啟用狀態、日期、時間、駕駛、油品、數量、單價 | 該車的加油 |
| 出勤原因、傷病患摘要、工作備註 | 工作 |
| 耗材內容或數量 | 該車的耗材 |
| 消毒方式、消毒項目 | 該車的消毒 |
| 單車／雙車切換、增加或移除車輛 | 依實際新增、移除或改動的車輛，套用上述規則 |

影響分析與執行規則：

- 只改一個站使用的資料時，只把該站改成 `*_needs_update`；其他已完成站保留。
- 雙車任務只改其中一車時，保留另一車既有的成功 checkpoint；補登只執行資料有變的車輛。
- 同一次編輯影響多個欄位時，受影響站別取聯集，但每站仍保留精確的受影響車輛。
- 只改空白、格式或順序，正規化後內容相同時，不產生補登影響，也不取消既有全站完成狀態。
- 已完成任務一旦產生有效的 `pending_edit_impact`，`all_complete` 立即變成 false；待受影響站別或車輛全部補登完成後，再由原子狀態調和恢復全站完成。
- 全站啟動按鈕重新出現時，只執行失敗、未完成或需更新的站別與車輛，已完成且資料未變的站別與車輛一律略過。
- 受影響的站別卡片顯示「更新工作／更新里程／更新加油／更新耗材／更新消毒」單站按鈕。
- 補登全部完成後，頁面不再顯示待處理的修改摘要；`task_updated` 事件仍保留欄位名稱、影響站別與車輛作為稽核紀錄。

安全邊界：

- 移除已經可能寫入官方網站的車輛，或取消已完成的加油紀錄時，不允許自動假設舊紀錄已消失。受影響站改為待人工確認，要求先到官方網站刪除或更新舊資料，再按既有具權杖的「已確認」。
- 若無法可靠判斷受影響車輛，採站別級補登，不保留該站的車輛 checkpoint。
- 編輯中的任務若正由本機流程或 Worker 執行，沿用既有鎖定規則，不允許同時修改。

## App 登打任務頁

### 尚未全部完成

頁面顯示目前進度及「四站／五站登打啟動」按鈕：

```text
進度：已完成 2/4
失敗：耗材、消毒

[ 四站登打啟動 ]  [ 返回編輯 ]
```

四／五站流程沿用既有行為：略過已完成站，執行未完成站。

### 完成後編輯資料

完成後若只修改第 2 車里程，頁面顯示：

```text
任務資料已修改
已修改：第 2 車里程
需重新登打：車輛里程（只重登第 2 車）

[ 更新里程 ]  [ 四站登打啟動 ]  [ 返回編輯 ]
```

「四站／五站登打啟動」仍保留原名稱，但此時只執行 `pending_edit_impact` 指定的站別與車輛。若只修改耗材，其他三站保持完成；若同時修改地址與耗材，則工作、里程、耗材三站顯示需更新。

### 最後一站補打後全部完成

頁面改顯示成功摘要：

```text
✓ 四站登打完成
最後完成：消毒（單站補打）

[ 返回編輯 ]
```

此時：

- 「四站／五站登打啟動」按鈕消失。
- 所有單站補打按鈕消失。
- 「返回編輯」保留。
- 執行紀錄保留「單站補打成功：消毒」。

按鈕是否顯示只看共用快照的 `all_complete`，不能只看 `overall_status`。待人工確認期間沿用現有安全限制，不顯示可重複啟動的登打按鈕。

## App 首頁最近任務

最近任務使用同一快照：

- 全部完成：進度文字顯示「四站登打完成」或「五站登打完成」，徽章顯示「完成」。
- 部分完成：顯示「已完成 3/4；失敗：消毒」等精確摘要。
- 單站成功但尚未全部完成：不得顯示完成徽章，也不得移入已完成任務分類。

## NAS 公務電腦後台

NAS 卡片不得再用最後一次操作事件直接代表整體狀態。卡片分成：

- **卡片主狀態**：由共用快照產生；全部完成時顯示「四站登打完成」或「五站登打完成」。
- **完整事件**：保留「四站登打成功」、「單站補打成功：消毒」、「人工確認站別完成」等歷史動作。
- **成功案件分類**：只有 `all_complete = true` 才能列入成功案件。

若最後一次操作是單站補打並使全站完成，卡片主狀態顯示：

```text
四站登打完成
```

展開完整事件仍可看到：

```text
單站補打成功：消毒
```

如此不需要製造第二筆假操作事件，也不會讓卡片標題停留在單站成功。

## 救護 Worker

Worker 每站仍保留各自的執行日誌。當 NAS 或本機 store 的原子調和首次回報 `all_complete = true` 時，Worker GUI 額外顯示一行：

```text
四站｜完成｜任務ID
```

有加油紀錄時顯示：

```text
五站｜完成｜任務ID
```

若單站補打成功但仍有未完成站，只顯示該站成功及剩餘進度，不得顯示四／五站完成。Worker 的文字是共用快照結果的呈現，不得另行以「這次執行沒有例外」推算全站完成。

## 異常與安全邊界

- 任務資料缺站、站別格式錯誤或有效站數無法判斷時，採未完成，不隱藏登打按鈕，也不歸入成功案件。
- 待人工確認仍是未完成；完成按鈕只能沿用既有具權杖的人工確認流程。
- 延遲、重複或過期 Worker 回報沿用 claim identity 與 status event ID 防護。
- 全站完成調和只改任務狀態與顯示，不重新開啟或送出任何官方網站表單。
- 不刪除既有失敗或補打事件，保留完整稽核時間線。

## 測試

先建立失敗回歸測試，再實作：

1. 單站成功、其餘三站未開始時：
   - `all_complete = false`。
   - 任務頁仍顯示「四站登打啟動」。
   - App 首頁與 NAS 後台不得顯示整筆完成或歸入成功案件。
2. 四站流程兩站成功、兩站失敗時：
   - 顯示「已完成 2/4；2 站失敗」。
   - 四站與單站補打入口仍可使用。
3. 補打一站後剩一站失敗時：
   - 顯示「已完成 3/4；失敗：站別」。
   - 不產生四站完成事件。
4. 最後一站單站補打成功時：
   - 整體原子轉成 `desktop_fast_completed`。
   - 任務頁顯示「四站登打完成」並隱藏所有登打按鈕。
   - NAS 卡片主狀態顯示「四站登打完成」，完整事件保留單站補打成功。
   - Worker 只新增一次「四站｜完成」日誌。
5. 啟用加油紀錄且五站全部完成時，所有表面顯示「五站登打完成」。
6. 完成後修改會影響任一站的資料時：
   - `all_complete` 變回 false。
   - 受影響站顯示需更新。
   - 頁面顯示「已修改」欄位、「需重新登打」站別及受影響車輛。
   - 登打按鈕重新出現。
7. 重複或延遲成功回報不得重複全站完成事件、通知或 Worker 完成日誌。
8. 完成後只修改耗材時：
   - 只有耗材變成 `consumables_needs_update`。
   - 工作、里程、消毒及未受影響的加油維持完成。
   - 「更新耗材」與全站啟動按鈕可用，執行時只跑耗材。
9. 雙車任務只修改第 2 車里程時：
   - 第 1 車里程 checkpoint 保留。
   - 第 2 車里程 checkpoint 清除並重新登打。
   - 工作、耗材、消毒與加油不受影響。
10. 修改案件地址時，只有工作與里程需更新；修改駕駛時，只有該車的工作、里程及已啟用加油需更新。
11. 正規化後資料沒有實質變化時，不建立 `pending_edit_impact`，既有完成狀態與按鈕隱藏狀態維持不變。
12. 移除已完成車輛或停用已完成加油時，受影響站進入待人工確認，不能自動標成完成。
13. 精準補登完成後：
   - `pending_edit_impact` 不再顯示為待處理。
   - 整體恢復「四站／五站登打完成」。
   - 全站與單站登打按鈕再次消失。

驗證範圍包含 `tests.test_task_store`、`tests.test_web_app`、`tests.test_desktop_fast_runner`、`tests.test_worker`、`tests.test_worker_gui`、完整 unittest、Python compile 與 `git diff --check`。
